Batches

By default the Eyes Web Applicationreports match results for every test.

In this screen shot you can see the Eyes Web Application after three tests have been run. TBD(link)

In some cases it is convenient to see the results of multiple tests together, as a single unit.

Eyes provides a way to group multiple tests together using batching.

There are two distinct scenarios where batching is required:

  • A test might be run in multiple execution environments , for example the tests may be written in different programing languages, or they may run concurrently on several machines, or at different times.
  • The same test may be run on different host environments, i.e. different browsers, operating systems or on mobile platforms. Since the test baseline depends on the host environment, each of these is reported as a separate test. The results of tests for all platforms can be collated and managed as a group by using batches.
  • From a software modularity point of view, different tests may be organized in individual modules/packages. Each of these is independent (i.e. each has it's own "Eyes" instance, and calls to open() with a unique test name. In this case, even if all the tests are run one after the other, they will be reported together but can be grouped together for the purpose of reviewing the results using batches.

Batching in the Eyes SDK

Batching in a single execution environment

If all the tests are run within a single execution then batching can be implemented as follows:

Java:

In Java, the test programer indicates that he wants to use batching by creating an instance of the class BatchInfobefore running any of the tests.

BatchInfo batchInfo = new BatchInfo("batch name");

Then later, in each test, before the eyes open() call is made, the programmer calls:

eyes.setBatch(batchInfo );

It can then pass this object to each test function and each tests calls setBatch() with this object.

Note that the string passed the BatchInfo constructor "batch name"is not the unique ID - it should be a name that is meaningful to the tester. Eyes generates internally a unique ID when the BatchInfo object is created. For this reason, when a batch is being compose din multiple execution runs a different technique needs to be used. This is described further here.
